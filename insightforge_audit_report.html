Critical Flaws

app/routers/training.py · L75, L84, L109, L121, L150 Critical · Correctness

Deprecated asyncio.get_event_loop() in Async Context

asyncio.get_event_loop() is deprecated since Python 3.10 inside a running async context. In Python 3.12 it emits a DeprecationWarning and in Python 3.14 it will raise a RuntimeError. The correct call is asyncio.get_running_loop(), which explicitly asserts an event loop is already running. The same bug appears in evaluation.py (4 occurrences). 

# ❌ Broken (training.py L75, evaluation.py L37)
loop = asyncio.get_event_loop()

# ✅ Fix
loop = asyncio.get_running_loop()
Fix Applied
app/routers/data.py · multiple endpoints Critical · Security (Path Traversal)

Raw Dataset Path Passed to os.path.exists()

Six endpoints in data.py pass request.dataset_path directly to os.path.exists() without sanitization. An attacker can supply ../../etc/passwd or any absolute path. The same pattern appears in training.py and evaluation.py. 

# ❌ Vulnerable
if not os.path.exists(request.dataset_path):
    raise HTTPException(...)

# ✅ Fix
SAFE_DIR = Path("datasets").resolve()
safe = (SAFE_DIR / Path(request.dataset_path).name).resolve()
if not safe.is_relative_to(SAFE_DIR) or not safe.exists():
    raise HTTPException(status_code=404, detail="Dataset not found")
Fix Applied
app/routers/evaluation.py · L91, L96 Critical · Disk Leak

Unbounded Joblib Files Saved During Every Evaluation Call

model_evaluation() saves a timestamped .joblib file to models/ for every model in every evaluation run. Under repeated usage the disk fills silently. Models are already persisted by the training step; re-saving during evaluation is redundant. 

# ❌ Saves new file per evaluation call
model_path = ml_agent.models_dir / f"{model_name}_tuned_{timestamp}.joblib"
joblib.dump(model, model_path)

# ✅ Fix: use a temp file, delete after evaluation
import tempfile, os
with tempfile.NamedTemporaryFile(suffix=".joblib", delete=False) as tmp:
    model_path = tmp.name
joblib.dump(model, model_path)
# ... evaluate ...
os.unlink(model_path)   # cleanup
app/routers/evaluation.py · L120–L240 Critical · Data Integrity

Fake /api/evaluation-report Returns Random Metrics

The legacy POST /api/evaluation-report endpoint generates results using random.uniform() — no dataset is touched. Any client calling this endpoint receives fabricated accuracy, precision, recall, and ROC-AUC values. The real endpoint is /api/evaluation-report-real. 

# ❌ Fake data
model_results[model][metric] = round(random.uniform(0.75, 0.95), 4)

# ✅ Fix: return deprecation notice
return JSONResponse(
    status_code=410,
    content={"deprecated": True,
             "warning": "This endpoint returns random data. Use /api/evaluation-report-real.",
             "redirect": "/api/evaluation-report-real"}
)
Fix Applied
app/agents/ml_training_agent.py · L222 Critical · Correctness

Hardcoded Titanic Column Names Dropped from Every Dataset

_preprocess_features() unconditionally drops ['PassengerId', 'Name', 'Ticket'] from any dataset. Any real-world dataset that happens to include a column named Name or Ticket (e.g., "Ticket Number", "Ticket ID") loses that column silently. 

# ❌ Hardcoded
id_columns = ['PassengerId', 'Name', 'Ticket']

# ✅ Fix: detect pure-ID columns heuristically
n = len(X)
id_columns = [
    col for col in X.columns
    if X[col].nunique() == n          # 100 % unique → identifier
    and (pd.api.types.is_integer_dtype(X[col])
         or pd.api.types.is_object_dtype(X[col]))
]
Fix Applied

High Severity Flaws

app/routers/training.py · L75, L84, L121 High · Resource Leak

Unbounded ThreadPoolExecutor (No max_workers)

Three ThreadPoolExecutor() instances are created inside request handlers with no max_workers cap. Python's default is min(32, os.cpu_count()+4). Under concurrent load each request spawns a fresh pool, leading to resource exhaustion. 

# ❌
with ThreadPoolExecutor() as executor:   # unlimited workers

# ✅ Module-level shared executor
_executor = ThreadPoolExecutor(max_workers=4)

# in handler:
loop = asyncio.get_running_loop()
result = await loop.run_in_executor(_executor, my_fn)
Fix Applied
app/routers/data.py · validate-target response High · Missing Data

validate-target Missing class_count Field

The POST /api/validate-target response omits class_count even though n_classes = target.nunique() is computed. The frontend store and Step 3 UI rely on this field to render class imbalance warnings correctly. 

# ❌ class_count computed but not returned
n_classes = target.nunique(dropna=True)
return {"is_valid": ..., "warnings": ..., ...}

# ✅
return {..., "class_count": int(n_classes)}
Fix Applied

Medium Severity Flaws

app/routers/analysis.py · feature_selection Medium · UX

is_classification Inferred from Heuristic, Not Provided task_type

Feature selection infers task type via y.dtype == "object" or len(y.unique()) < 20. The caller already knows the task type (it was set in Step 3). Passing task_type as a request parameter would be more reliable and avoid misclassification for numeric binary targets with many classes.

app/agents/ml_training_agent.py Medium · UX

No Training Progress Streaming

Model training can take 30–120 seconds. There is no SSE/WebSocket progress feed during actual training (only for hyperparameter tuning). The frontend spinner provides no meaningful progress indication.

app/routers/evaluation.py Medium · Reliability

No Model Count / Disk Space Guard Before Training

With no cleanup policy, models/ accumulates .joblib files indefinitely. A MAX_MODELS = 20 eviction policy (LRU by mtime) would bound disk usage automatically.

Low Severity & Frontend Issues

Low · Frontend

localStorage Overflow Risk

The partialize function strips large arrays but still persists profileResult, edaResult, and featureEngineeringResult which can be several hundred KB each. Browsers typically cap localStorage at 5 MB. Strip these from persistence entirely — re-fetch on page reload from session state.

Low · Developer Experience

No Structured Logging or Request ID

Errors are logged via logger.error(str(e)) with no request ID, endpoint name, or user context. Add a middleware that attaches a UUID trace ID to each request and includes it in every log line.

Low · Developer Experience

No API Rate Limiting

Endpoints like /api/model-training are computationally expensive. Without rate limiting, a single client can submit dozens of concurrent training jobs and exhaust server RAM. Add slowapi or fastapi-limiter.

Low · Reliability

No Background Job Queue for Long-Running Tasks

Training and hyperparameter tuning are executed synchronously inside request handlers. If the client disconnects mid-request, the computation still runs but results are lost. Consider Celery + Redis or FastAPI BackgroundTasks with a proper job-status polling endpoint.

Low · Frontend

No Offline / Error Recovery State

If the backend returns a 500 during a mid-pipeline step, the frontend marks the step as failed but has no retry button. The user must reload and lose all prior context. Add per-step retry actions and persist the last successful API response for each step.

Security Concerns (OWASP)

A01 — Broken Access Control

No Authentication on Any Endpoint

Every API endpoint is fully public. Anyone who can reach the server can upload datasets, train models, and read reports. Add JWT / API-key middleware as a minimum. For multi-user scenarios, scope datasets and reports to an authenticated identity.

A05 — Security Misconfiguration (Path Traversal)

Raw Filesystem Paths Accepted from Clients

As noted in Critical Flaws: request.dataset_path is passed to os.path.exists() without canonicalization. This is an OWASP A05 violation. Always resolve paths against a safe base directory.

A08 — Software and Data Integrity Failures

Fake Evaluation Metrics Served as Real

The legacy /api/evaluation-report endpoint returns random.uniform() values branded as model metrics. This violates data integrity — users making model selection decisions based on these values are misled.

Suggested Feature Additions

📊 SHAP Explainability Panel

Post-training SHAP waterfall / beeswarm plots using shap library. Show per-prediction explanations in Step 9 UI.

🔁 Automated Retraining on Drift

Integrate Evidently AI or custom PSI checks. Alert or auto-retrain when input distribution shifts beyond a threshold.

📁 Project / Experiment History

Persist experiment metadata (dataset, model, metrics, date) in SQLite via SQLModel. Allow browsing past runs and comparing metrics.

🧪 A/B Model Comparison

Let users pin two model evaluation results side-by-side with radar charts and McNemar statistical significance tests.

📤 One-Click Export

Export trained model as ONNX, Python inference script, or Docker image from the UI.

🤖 LLM Auto-Pilot Mode

Full pipeline auto-run: LLM chooses cleaning strategies, feature engineering, model selection, and tuning budget autonomously.

🗃️ Data Versioning

Hash-based dataset versioning so each pipeline step references an immutable snapshot. Enables reliable re-runs and diff views.

⏱️ Real-Time Training Progress

SSE stream from the training endpoint pushing epoch/trial progress. Replace static spinner with live loss/metric chart.

🔑 Multi-User Auth + Workspaces

JWT authentication with user-scoped dataset/model namespaces. Each user only sees their own projects.

📋 Interactive Prediction Playground

After training, expose a form where users enter feature values and see real-time predictions with confidence intervals.

📈 AutoML Benchmark Mode

Run all supported models in one click with shared CV folds. Produce a leaderboard ranked by cross-validated score.

📦 Plugin / Custom Model Upload

Allow users to upload a scikit-learn-compatible .py estimator and have it appear in the model selection list.
